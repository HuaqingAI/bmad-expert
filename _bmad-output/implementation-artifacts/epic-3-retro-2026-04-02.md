## 做得好的地方（Keep）

### 1. fixSteps 的向后兼容设计
`BmadError` 的第四参数 `fixSteps = []` 有默认值，所有 Epic 1/2 中的现有调用（`new BmadError(code, message, cause)`）无需任何修改即可兼容——这是正确的增量增强方式，零破坏性。

### 2. AI 可自愈的错误消息设计（E004）
E004 的 fixSteps 包含：
1. `手动创建并授权目标目录：mkdir -p ~/.happycapy/agents/bmad-expert`
2. `确认路径权限后重新执行：npx bmad-expert install`

这两条指令 AI 可以直接解析并执行，无需人工介入——这是"AI 可自愈"设计目标的直接体现。

### 3. Node.js 版本检查的职责边界清晰（Story 3.2）
版本检查代码放在 `cli.js` 而非 `lib/` 模块——符合架构约束（lib 模块不允许 `process.exit`），且逻辑清晰：CLI 入口处早期失败，避免版本问题在安装流程深处爆出。

### 4. wrapNetworkError 与 wrapFileError 同构设计
Story 3.3 的 `wrapNetworkError` 与 3.1 的 `wrapFileError` 完全同构：检测错误码 → 包装为对应 BmadError → 传入 fixSteps。这种一致的包装函数模式使错误处理代码可预测、可测试。

### 5. 跨 Story 修改测试的显式记录
Story 3.1 更新了 Epic 2 的 `test/installer.test.js` 中两条 E004 测试的断言（匹配新的 cause 格式），且在 File List 中明确列出了被修改的文件。这种跨 Epic 的代码修改需要显式记录。

## 需要改进的地方（Change）

### 1. Story 3.3 是"预留场景基础设施"，当前无实际触发路径
**现象：** 当前 bmad-expert 的所有 agent 文件内嵌在 npm 包内，安装过程完全不需要网络请求。E005 和 `wrapNetworkError` 在当前版本中永远不会被真实触发。
**影响：** 工作量投入在一个假想的未来场景，优先级上可能有更高价值的事情未完成（如 Epic 5 文档）。
**建议：** "预留基础设施" Story 在创建时应明确标注，便于在优先级排序时做知情决策。

### 2. Epic 2 测试被 Epic 3 修改，范围扩散
**现象：** Story 3.1 更新了 `test/installer.test.js`（Epic 2 的测试）中的断言。这是合理的（BmadError 行为变化导致测试需更新），但"修改其他 Epic 的测试"需要明确的记录和注意。
**建议：** 在 story 的 Dev Notes 中提前标识"本 Story 会破坏 XYZ 测试的断言"，帮助 dev 在实现时主动扫描。

### 3. Story 文件状态持续不一致（第三次出现）
3.2 和 3.3 的文件状态为 `review`，3.1 为 `done`。此问题在 Epic 1、2 均已出现，Epic 3 仍未改进。这证明需要流程层面的保障，而非依赖个人自觉。

## 关键经验总结

1. **错误设计的价值在于 AI 可以读懂并执行**：fixSteps 不是"给人看的说明文字"，而是"AI 可以直接执行的命令序列"——这个设计意图在 Story 3.1 中体现得很清晰。
2. **向后兼容的增量增强**：在 BmadError 上添加可选参数是正确做法，相比"重构 BmadError 接口"节省了大量更新现有调用点的工作。
3. **预留基础设施需要有意识地管理**：Story 3.3 是合理的技术投资，但需要知道"这是预留的"，避免后续将其误认为已被测试的完整功能。
