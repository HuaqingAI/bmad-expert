## 做得好的地方（Keep）

### 1. Quick Start 放在顶部——"新用户第一眼"原则
重写后的 README 将 `## Quick Start` 放在 badge 之后、介绍之前，确保新用户打开文档第一眼看到的是"怎么用"，而不是背景介绍。这是正确的用户优先设计。

### 2. 触发命令使用自然语言格式（FR32）
HappyCapy 平台的触发命令设计为可直接粘贴到聊天框的自然语言句子（`Please run npx bmad-expert install to install BMAD Expert`），而非 shell 命令——符合 AI 平台的使用模式，降低了技术门槛。

### 3. 为未来平台预留扩展结构
README 使用 `### HappyCapy` + `### Claude Code（即将支持）` + `### OpenClaw（即将支持）` 的章节结构，Phase 1.5 添加新平台时只需追加子章节，无需重构。

### 4. 删除开发者临时内容（npm 包发布说明）
Story 5.1 明确要求删除原 README 中面向开发者的"npm 包发布说明"。这类开发期临时提醒不应出现在面向用户的文档中——及时清理保持了 README 的专注性。

### 5. 需求规格详细（8 条可验证检查清单）
Story 5.1 的"实现检查清单"包含 8 条可直接验证的具体标准（Quick Start 章节、术语解释、步骤数量、平台扩展结构等），使 README 文档类 Story 的验收不再依赖主观判断。

## 需要改进的地方（Change）

### 1. Epic 5 在 Epic 6 之后完成——文档欠债
**现象：** README 的 Quick Start 只包含 `install` 命令的触发方式，但 Epic 6 已交付了 `update`、`status`、`--json` 三个新命令，均未在 README 中有任何描述。
**影响：** 外部用户或 AI 调用方无法从文档发现这三个功能，尤其是 `--json` 模式（专为 AI 调用设计）没有文档，等于功能对外不可见。
**根因：** Epic 5 和 Epic 6 的顺序反转（Growth 功能优先于文档），加之 Epic 6 回顾中未将 README 更新纳入 Epic 6 的 AC 范围。

### 2. Story 5.1 的文件状态字段是迄今最严重的不一致
**现象：** Story 5.1 的 `Status` 字段值为 `ready-for-dev`，而 sprint-status 显示 `epic-5: done`。
**含义：** 这意味着 story 文件在开始实现之前就被创建，但实现完成后状态字段甚至没有从 `ready-for-dev` 更新到 `review` 或 `done`。
**影响：** 与 Epic 1-4 的 `review` 状态遗留不同，`ready-for-dev` 给读者的错误信号更强——看起来像是"还未开始实现"的 Story。

### 3. README 覆盖截止 Epic 4，有意识的信息截止线不够清晰
Story 5.1 的"已实现功能"部分只到 Epic 4，Epic 3 也只标注 "3-1 完成（权限拒绝错误）"——其实 Epic 3 全部 3 个 Story 都已完成。这说明 Story 创建时的上下文快照不够准确。

## 关键经验总结

1. **文档欠债随功能扩展快速累积**：Epic 6 在 Epic 5 之后完成，但 Epic 5 的 README 仅覆盖 install，不覆盖 update/status/--json。这说明功能和文档必须同步交付，或者功能 Epic 的 AC 中必须包含文档更新要求。
2. **故事文件状态是团队健康指标**：从 Epic 1 到 Epic 5，story 文件状态不一致问题一直存在且未改善。这不是单个 dev 的疏忽，而是缺少强制性流程保障的系统问题。
3. **文档类 Story 的可验证检查清单有效**：Story 5.1 的 8 条检查清单使文档验收有了客观标准——这个模式可以推广到所有文档类 Story。
